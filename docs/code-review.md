# Mermaid ER Viewer コードレビュー

## 総合評価: よくできた個人開発ツール

MermaidのER図をインタラクティブに表示・編集し、MCPサーバーとしても動作するという着眼点は非常に良い。約3,600行のTypeScriptで、パーサー・サーバー・クライアント・MCP全てを自前で実装しており、コンパクトにまとまっている。

---

## 良い点

### 設計判断が的確
- .mmdファイルは読み取り専用、レイアウトは .layout.json サイドカーファイルに分離 — Mermaid標準準拠と独自メタデータの共存が綺麗
- WebSocketによるリアルタイム同期。外部編集（他のAI等）にも即座に反応する
- ビューポートカリングによるパフォーマンス最適化
- Sugiyama/Louvain系のオートレイアウトは、この規模のツールとしては本格的

### コードの品質
- TypeScript strict mode、型定義が整っている
- 関心の分離が明確（parser / server / client が独立）
- ドラッグ&ドロップのrAFスロットリング、テキスト計測キャッシュなど細かい最適化がされている
- Undo/Redo実装、コンパクトモード切替のキャンバス状態保持など、UX配慮が行き届いている

### MCP統合
- parse-diagram, get-layout, save-layout, open-viewer, close-viewer と、Claude Codeからの操作に必要十分なツールが揃っている
- プロセス終了時の自動クリーンアップも配慮されている

---

## 問題点・指摘

### 1. テストが一切ない
最大の弱点。パーサー、レイアウトアルゴリズム、LayoutStore等はユニットテスト可能で、リグレッション防止に不可欠。特にパーサーは正規表現ベースなのでエッジケースの検証が重要。

### 2. セキュリティ: パストラバーサル
`server.ts:103-104` の `/api/browse` と `server.ts:161` の `/api/open` は、ユーザー入力の `path` をそのまま `resolve()` して任意のファイルシステムにアクセス可能。`baseDir` 配下に制限するチェックがない。

```typescript
// 現状: 任意のパスにアクセスできる
const targetPath = req.query.path ? resolve(String(req.query.path)) : baseDir;
```

### 3. パーサーの限界
- `ATTRIBUTE_RE` がアンダースコアなしの型名（`varchar(255)` のようなパラメータ付き型）をパースできない
- Mermaid公式パーサーとの互換性検証がない。`erDiagram`の後にタイトルやアクセシビリティ宣言がある場合を無視する
- 正規表現の `{1,2}` では `}o`（2文字）は捕まるが、`}`（1文字単体）もマッチしてしまう可能性がある

### 4. `main.ts` が肥大化（815行）
イベントハンドリング、状態管理、ハイライトロジック、ラベル編集UIが1ファイルに集中。保守性に影響する。

### 5. PUT /api/layout のバリデーション不足
`server.ts:271` で `req.body as LayoutData` とキャストしているだけで、スキーマバリデーションなし。不正なJSONを送られるとサイレントに壊れる。

### 6. layoutSavingの競合制御がflaky
`server.ts:275` の `setTimeout(() => { layoutSaving = false; }, 500)` は、ファイル監視のタイミングに依存するハック。書き込みが遅いディスクや高負荷環境で競合する可能性がある。

### 7. `render()` が毎回 `innerHTML = ''` で全再構築
`renderer.ts:152-153` で毎回エンティティとコネクタを全消去して再構築。差分更新にすればスクロール時のちらつきを防げる。

## 改善提案

### 優先度高

| 提案 | 理由 |
|------|------|
| **パーサーのユニットテスト追加** | vitest導入で30分程度。パーサーは独立性が高くテストしやすい |
| **パストラバーサル防止** | `resolve(path).startsWith(baseDir)` のチェック1行で修正可能 |
| **PUT /api/layout にzodバリデーション** | zodは既にMCP側で依存しているので、サーバー側でも使える |

### 優先度中

| 提案 | 理由 |
|------|------|
| **エクスポート機能 (SVG/PNG)** | ER図ビューワーとして最も需要が高い追加機能 |
| **検索/フィルタ** | エンティティ数が多い場合に特定テーブルを素早く見つけたい |
| **ダークモード/ライトモード切替** | 現状はTokyoNightダーク固定。ライトモードの需要は高い |
| **タッチ操作対応** | タッチイベントが未対応。iPadなどで使えない |
| **main.ts の分割** | ハイライト関連、ラベル編集、パン/ズーム管理を個別モジュールに |

### 優先度低（差別化のアイデア）

| 提案 | 理由 |
|------|------|
| **ミニマップ** | 大規模ER図でのナビゲーション補助。SVGのサムネイルをコーナーに表示 |
| **エンティティのグループ化/色分け** | layout.jsonに `groups` フィールドを追加し、アプリ/ドメイン単位で色分け |
| **.mmd差分ハイライト** | ファイル変更時に追加/削除されたエンティティ・カラムを一時的にハイライト |
| **複数ファイル同時表示** | マイクロサービス等で分割されたER図を統合ビューで見たい |
| **MCPツール: auto-layout** | Claude Codeからレイアウトを自動整列できるツール |

---

## まとめ

コードの構造は良くまとまっており、MCPサーバー統合という独自の価値がある。最も急務なのは**テストの追加**と**パストラバーサル修正**の2点。機能面では**エクスポート**と**検索**が最もユーザー価値を高めるだろう。
